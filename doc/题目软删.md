现在项目需要继续增量开发一个新功能：**题目删除 / 恢复 / 重复题处理预留**。

当前系统已经支持：

* 题库导入
* 题目编辑
* 答案纠错
* 修改历史
* 练习模式
* 错题复习
* Attempt 快照
* Practice Session

现在发现一个实际问题：

> 有些题目可能本身不对、格式错误、重复、已经不适合继续练习。系统目前不允许删除题目，导致这些题仍然会出现在题库、练习、错题、统计中。

请在现有架构上做增量开发，不要重写项目。

## 一、核心原则

题目删除不要优先做物理删除。

请优先实现：

```text
软删除 soft delete
```

也就是题目仍然保留在数据库里，但默认不再出现在：

* 题库列表
* 随机练习
* 顺序练习
* 按方向练习
* 按考察点练习
* 未答题练习
* 错题专项练习
* 普通统计

这样可以避免历史数据断裂。

## 二、为什么不能直接硬删除

当前系统已经有：

```text
attempts
question_revisions
practice_sessions
wrong question logic
stats
```

如果直接物理删除 `questions` 表记录，可能导致：

1. 历史答题记录找不到题目
2. 错题记录找不到题目
3. 修改历史无法追溯
4. Practice Session 里残留已删除 question_id
5. 统计接口异常
6. 后续恢复困难
7. 重复题判断和题库审计困难

所以本阶段只做软删除。
物理删除可以预留，但不要作为默认功能。

## 三、新增字段

给 `questions` 表增加：

```python
is_deleted: bool = False
deleted_at: datetime | None = None
delete_reason: str | None = None
deleted_source: str | None = None
```

字段说明：

* `is_deleted`：是否已删除
* `deleted_at`：删除时间
* `delete_reason`：删除原因
* `deleted_source`：删除来源，默认 `manual_delete`

如果当前数据库已经存在，需要做轻量迁移，不要让我删库重建。

所有新增字段都要兼容已有数据：

```text
已有题目默认 is_deleted = false
```

## 四、删除也必须留痕

删除题目不能无记录。

可以复用现有 `question_revisions` 表，也可以新增 `QuestionDeletionLog`。

优先复用 `question_revisions`。

删除时生成一条 revision：

```text
changed_fields = ["is_deleted", "deleted_at", "delete_reason"]
source = "manual_delete"
reason = 用户填写的删除原因
```

恢复时也生成 revision：

```text
changed_fields = ["is_deleted", "deleted_at", "delete_reason"]
source = "restore_deleted_question"
reason = 用户填写的恢复原因
```

## 五、后端 API

### 1. 软删除题目

新增：

```http
DELETE /api/questions/{question_id}
```

请求体：

```json
{
  "reason": "题目重复，已删除"
}
```

逻辑：

1. 检查题目是否存在
2. 如果已经删除，返回明确提示
3. 设置 `is_deleted = true`
4. 设置 `deleted_at`
5. 设置 `delete_reason`
6. 设置 `deleted_source = manual_delete`
7. 生成 revision
8. 不删除 attempts
9. 不删除 question_revisions
10. 不删除 practice_sessions
11. 返回删除后的题目状态

返回：

```json
{
  "question_id": "xxx",
  "is_deleted": true,
  "deleted_at": "...",
  "delete_reason": "题目重复，已删除"
}
```

### 2. 恢复已删除题目

新增：

```http
POST /api/questions/{question_id}/restore
```

请求体：

```json
{
  "reason": "误删，恢复题目"
}
```

逻辑：

1. 检查题目是否存在
2. 检查题目是否已删除
3. 设置 `is_deleted = false`
4. 清空 `deleted_at`
5. 清空或保留 `delete_reason`，建议清空并在 revision 中保留历史
6. 生成 revision
7. 返回恢复后的题目

### 3. 获取已删除题目列表

新增：

```http
GET /api/questions/deleted
```

支持分页：

```text
page
page_size
keyword
type
difficulty
direction
exam_point
```

返回：

```json
{
  "items": [],
  "total": 12,
  "page": 1,
  "page_size": 20,
  "has_next": false
}
```

### 4. 题目列表支持 include_deleted

现有：

```http
GET /api/questions
```

增加参数：

```text
include_deleted=false
only_deleted=false
```

默认：

```text
include_deleted=false
only_deleted=false
```

也就是默认不展示已删除题目。

如果：

```text
only_deleted=true
```

则只查已删除题目。

### 5. 题目详情支持查看已删除题目

```http
GET /api/questions/{question_id}
```

即使题目已删除，也可以查看详情，但返回中需要包含：

```json
{
  "is_deleted": true,
  "deleted_at": "...",
  "delete_reason": "..."
}
```

前端展示明显的“已删除”状态。

4. 不应该再出现在错题专项里
5. 不应该再出现在未答题里
6. 不应该再进入统计中的总题数

## 七、已有 Practice Session 的处理

如果用户之前创建的 Practice Session 里包含后来被删除的题目，需要处理：

### 推荐逻辑

在读取 session 当前题目时，如果发现 question 已删除：

```text
自动跳过该题，进入下一题
```

如果该组里有已删除题目：

```text
前端不展示已删除题目
```

如果整个 session 剩余题都被删除：

```text
提示：当前练习中的题目已被删除或不可用，请重新创建练习。
```

后端需要避免返回空对象或 500 错误。

## 八、错题页处理

错题页默认不展示已删除题目。

如果用户确实要查看历史，可以后续增加：

```text
显示已删除错题
```

但本阶段默认：

```text
错题列表排除 is_deleted = true 的题目
```

原因：

* 用户删除题目通常说明题目不应该继续复习
* 错题专项不应该继续练已删除题

历史 attempts 仍然保留，不删除。

## 九、统计处理

统计页默认排除已删除题目。

包括：

* 总题数
* 已答题数
* 正确率
* 错题数
* 方向统计
* 考察点统计
* 未答题统计

但是历史 attempts 不删除。

如果统计逻辑需要 join questions，请加：

```text
questions.is_deleted = false
```
