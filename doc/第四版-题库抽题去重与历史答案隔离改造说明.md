# 题库抽题去重与历史答案隔离改造说明

> 目标读者：Codex / 项目开发者  
> 改造目标：解决题库系统中“下一批题重复抽到已答题目”和“已答题目的历史答案被原封不动显示在新一轮答题页”的问题。  
> 改造原则：不重写现有架构；在现有题目、Attempt、导入顺序、随机抽题、错题/提醒功能基础上做增量改造。

---

## 1. 当前问题描述

当前题库系统存在以下问题：

1. 用户完成第一批题后，进入第二批题时，系统仍可能抽到第一批已经答过的题。
2. 被重复抽到的题目会把用户上一次选择的答案直接显示出来。
3. “新题练习”“错题复习”“历史复盘”三种行为没有清晰隔离。
4. 随机抽题、顺序抽题、提醒/复习抽题之间缺少统一的候选池过滤规则。
5. 当剩余未答题数量不足时，系统可能隐式混入已答题，而没有明确提示用户。

这些问题会导致：

- 新题练习被历史作答污染；
- 用户无法稳定刷下一批未完成题；
- 答题页和复盘页职责混乱；
- 错题复习和重复抽题无法区分；
- 题库进度统计不准确。

---

## 2. 参考成熟系统的设计思路

本系统可以参考以下成熟答题/学习系统的抽题思想：

### 2.1 Moodle Quiz

Moodle 的随机题不是简单从全题库随机，而是从题库分类、标签、题组等范围内抽题。它明确区分：

- 题库中的题；
- 一次 quiz attempt 中实际出现的题；
- 多次 attempt 的评分与反馈；
- 是否立即展示答案、解析、反馈。

关键启发：

- 同一次练习/测验中，不应出现重复题。
- 随机题需要从指定候选池里抽，而不是全表随机。
- 如果候选池题目数量不足，应提示错误或让用户选择处理方式，而不是静默重复。
- 答案、正确性、解析的展示时机应该由 review policy 控制。

### 2.2 Canvas Quiz / Question Group

Canvas 的 question group 思路是：先定义一组候选题，再指定从这组里随机抽几道题。

关键启发：

- 抽题应该基于“候选题组 / 条件过滤结果”。
- 随机只是排序策略，不应该绕过业务过滤规则。
- 一个练习批次生成后，题目列表应该固定下来，不能刷新一次重新随机一次。

### 2.3 Anki

Anki 不把所有卡片混在一起，而是区分：

- New：新卡；
- Learn：学习中；
- Due / Review：到期复习；
- Suspended / Buried：暂停或临时隐藏。

关键启发：

- “新题”和“复习题”应该分池管理。
- 错题/到期复习应有单独入口。
- 同一轮学习中，可以临时隐藏不该重复出现的题。
- 历史答案是复盘信息，不是下一轮作答的默认输入。

## 3. 本次改造的核心结论

本次不是简单修改随机算法，而是要补齐三个核心能力：

1. **用户题目状态表**  
   记录每个用户对每道题的学习状态，例如是否答过、是否答对、是否错题、是否已掌握、下次复习时间。

2. **练习批次表**  
   每次生成一批题时，把题目固定到一个 session 中。用户刷新、继续答题、返回答题，都读取同一个 session，不重新随机。

3. **答题页与复盘页隔离**  
   新一轮答题不得预填历史答案。历史选择、正确答案、解析只能在提交后、复盘页或用户主动展开历史时展示。

一句话原则：

> 题目是否能被抽中，由 `user_question_state` 和 `practice_session` 控制；答案是否能展示，由当前 attempt 状态和 review policy 控制。不要把历史 attempt 当成本次 attempt。

## 4. 术语定义

### 4.1 Question

题库中的题目定义。

建议继续使用现有 `questions` 表，不要重写。

核心字段示例：

```text
questions
- id
- question_id / external_id
- module
- type
- difficulty
- knowledge_points
- stem
- options
- answer
- explanation / option_analysis
- import_order
- status              # draft / ready / disabled / deleted
- version
- created_at
- updated_at
```

### 4.2 Attempt

一次具体作答记录。

Attempt 代表“用户某一次对某题的作答”，不是题目当前状态。

建议字段：

```text
attempts
- id
- user_id
- question_id
- question_version
- question_snapshot
- practice_session_id
- selected_answer      # 本次作答答案；新建 attempt 时必须为空
- answer_text           # 主观题答案；新建 attempt 时必须为空
- is_correct
- score
- status               # not_started / answering / submitted / graded
- started_at
- submitted_at
- created_at
- updated_at
```

关键要求：

- 新建 attempt 时，`selected_answer` / `answer_text` 必须为空。
- 禁止从历史 attempt 复制答案到新的 attempt。
- 历史答案只能通过复盘接口读取。

### 4.3 UserQuestionState

每个用户对每道题的累计学习状态。

这是本次改造最重要的新表。

```text
user_question_state
- id
- user_id
- question_id
- current_question_version
- status                # new / attempted / correct / wrong / reviewing / mastered / suspended
- attempt_count
- correct_count
- wrong_count
- consecutive_correct_count
- consecutive_wrong_count
- last_attempt_id
- last_attempt_at
- last_result            # correct / wrong / partial / ungraded
- last_answer_snapshot   # 可选，用于复盘，不用于新答题页预填
- next_review_at         # 到期复习时间
- mastery_level          # 0-5 或 0-100
- is_favorited
- is_marked
- is_excluded
- created_at
- updated_at
```

唯一索引：

```sql
CREATE UNIQUE INDEX uq_user_question_state_user_question
ON user_question_state(user_id, question_id);
```

### 4.4 PracticeSession

一次练习批次。

用户每点击“开始答题 / 下一组题 / 随机 20 题 / 顺序 20 题”，都应该创建或恢复一个 practice session。

```text
practice_session
- id
- user_id
- mode                 # new_random / new_order / wrong_review / due_review / all_practice / exam / review_history
- module
- batch_size
- requested_count
- actual_count
- status               # active / completed / abandoned
- allow_answered        # 是否允许混入已答题
- show_history_policy   # never / after_submit / manual_expand / always_review_only
- created_at
- completed_at
```

### 4.5 PracticeSessionQuestion

固定一次练习批次中的题目列表。

```text
practice_session_question
- id
- practice_session_id
- question_id
- question_version
- order_index
- source_reason         # new / wrong / due_review / all_practice / manual
- created_at
```

唯一索引：

```sql
CREATE UNIQUE INDEX uq_session_question
ON practice_session_question(practice_session_id, question_id);
```

## 5. 答题模式设计

前端应把模式拆清楚，不要混在一个“随机/顺序/提醒”开关里。

| 模式 | mode | 抽题范围 | 排序方式 | 是否允许已答题 | 是否显示历史答案 |
|---|---|---|---|---|---|
| 顺序新题 | `new_order` | 未答题 | `import_order ASC` | 否 | 否 |
| 随机新题 | `new_random` | 未答题 | 随机 | 否 | 否 |
| 错题复习 | `wrong_review` | 错题 / 未掌握题 | 最近错误优先或随机 | 是，仅错题 | 不预填，可提交后显示 |
| 到期复习 | `due_review` | `next_review_at <= now` | 到期时间优先 | 是，仅到期题 | 不预填，可提交后显示 |
| 收藏练习 | `favorite` | 收藏题 | 可选 | 是 | 不预填 |
| 全量练习 | `all_practice` | 全部 ready 题 | 可选 | 是 | 默认不预填 |
| 考试模式 | `exam` | 按规则抽题 | 固定批次 | 否或按考试规则 | 严禁显示 |
| 历史复盘 | `review_history` | 历史作答记录 | 时间倒序 | 不涉及 | 显示 |

核心要求：

- 默认练习入口应使用 `new_order` 或 `new_random`。
- 只有用户明确选择“错题复习 / 全量练习 / 复盘”时，才允许已答题进入候选池。
- 新 attempt 永远不预填历史答案。

## 6. 抽题规则

### 6.1 统一候选池构建流程

所有抽题模式都应该走同一个候选池构建函数，例如：

```python
def build_candidate_query(user_id, mode, module=None, filters=None):
    """
    返回符合当前模式的题目候选集。
    注意：这里只构造候选池，不负责最终排序和 limit。
    """
```

通用过滤条件：

```text
1. question.status = 'ready'
2. 如果传入 module，则 question.module = module
3. 如果传入 difficulty / type / knowledge_points，则继续过滤
4. 排除被用户 is_excluded 的题
5. 排除当前 session 已存在的题
6. 根据 mode 决定是否排除已答题
```

### 6.2 顺序新题模式

规则：

```text
- 只抽用户未答题。
- 按导入顺序 import_order ASC。
- 已答题不再出现。
```

SQL 逻辑示例：

```sql
SELECT q.*
FROM questions q
LEFT JOIN user_question_state uqs
  ON q.id = uqs.question_id
 AND uqs.user_id = :user_id
WHERE q.status = 'ready'
  AND (:module IS NULL OR q.module = :module)
  AND (uqs.id IS NULL OR uqs.status = 'new')
ORDER BY q.import_order ASC, q.id ASC
LIMIT :batch_size;
```

### 6.3 随机新题模式

规则：

```text
- 只抽用户未答题。
- 从未答题池中随机。
- 已答题不再出现。
```

SQLite 示例：

```sql
SELECT q.*
FROM questions q
LEFT JOIN user_question_state uqs
  ON q.id = uqs.question_id
 AND uqs.user_id = :user_id
WHERE q.status = 'ready'
  AND (:module IS NULL OR q.module = :module)
  AND (uqs.id IS NULL OR uqs.status = 'new')
ORDER BY RANDOM()
LIMIT :batch_size;
```

PostgreSQL 示例：

```sql
ORDER BY RANDOM()
```

MySQL 示例：

```sql
ORDER BY RAND()
```

如果题量较大，不建议直接 `ORDER BY RANDOM()` 全表随机。可以后续优化为：

```text
1. 先查候选题 id；
2. 在应用层随机采样；
3. 再按采样 id 回表查询题目；
4. 或使用预生成 random_key 字段。
```

当前阶段如果题量不大，可以先使用数据库随机排序。

### 6.4 错题复习模式

规则：

```text
- 只抽 status = wrong / reviewing 的题。
- 不预填上次答案。
- 可以在题目卡片上显示“上次错误 / 已错 2 次”等状态标签。
- 提交本次答案后，才展示正确答案、解析、历史选择。
```

SQL 示例：

```sql
SELECT q.*
FROM questions q
JOIN user_question_state uqs
  ON q.id = uqs.question_id
WHERE q.status = 'ready'
  AND uqs.user_id = :user_id
  AND uqs.status IN ('wrong', 'reviewing')
  AND (:module IS NULL OR q.module = :module)
ORDER BY uqs.last_attempt_at DESC
LIMIT :batch_size;
```

### 6.5 到期复习模式

规则：

```text
- 只抽 next_review_at <= now 的题。
- 适合做提醒/复习入口。
- 不应该和新题入口混合。
```

SQL 示例：

```sql
SELECT q.*
FROM questions q
JOIN user_question_state uqs
  ON q.id = uqs.question_id
WHERE q.status = 'ready'
  AND uqs.user_id = :user_id
  AND uqs.next_review_at IS NOT NULL
  AND uqs.next_review_at <= :now
  AND uqs.status IN ('wrong', 'reviewing', 'attempted')
  AND (:module IS NULL OR q.module = :module)
ORDER BY uqs.next_review_at ASC
LIMIT :batch_size;
```

### 6.6 全量练习模式

规则：

```text
- 可以抽全部 ready 题。
- 允许已答题进入候选池。
- 但仍然不能预填历史答案。
- 前端可以显示“已做过 / 上次正确 / 上次错误”状态标签。
```

用途：

```text
- 用户主动想重新刷全部题。
- 题库量较小，用户想反复训练。
- 面试前快速过一遍所有题。
```

### 6.7 题目数量不足处理

如果用户请求 20 道新题，但未答题只剩 8 道，不能静默混入已答题。

后端应该返回明确状态：

```json
{
  "code": "NOT_ENOUGH_NEW_QUESTIONS",
  "requested_count": 20,
  "available_count": 8,
  "options": [
    "use_available_only",
    "include_wrong_questions",
    "include_answered_questions",
    "reset_progress"
  ]
}
```

前端提示：

```text
当前模块未答题仅剩 8 道，不足 20 道。

你可以选择：
1. 只生成剩余 8 道新题
2. 切换到错题复习
3. 允许混入已答题
4. 重置该模块答题进度
```

只有用户明确选择“允许混入已答题”时，才设置：

```text
allow_answered = true
```

## 7. 答案展示规则

### 7.1 新答题页禁止预填历史答案

无论题目是否做过，只要当前页面是“答题页”，新建 attempt 时必须：

```text
selected_answer = null
answer_text = null
status = not_started / answering
```

禁止做法：

```text
读取 last_attempt.selected_answer → 赋值给当前 attempt.selected_answer
```

正确做法：

```text
读取 last_attempt 只能用于展示状态标签或复盘入口，不得用于当前答案字段。
```

### 7.2 答题页可以显示的历史状态

允许显示：

```text
- 未答过
- 已答过 1 次
- 上次正确
- 上次错误
- 正确率 60%
- 已收藏
- 已标记
```

禁止显示：

```text
- 上次选择了 B
- 正确答案是 C
- 解析内容
- option_analysis
```

除非当前模式是 `review_history` 或当前 attempt 已提交。

### 7.3 提交后展示

用户提交当前 attempt 后，可以展示：

```text
- 本次选择
- 正确答案
- 本次是否正确
- 解析
- 选项逐项分析
- 历史作答次数
- 上次作答结果
```

但注意：

```text
提交后展示的是结果反馈，不是新答题预填。
```

### 7.4 复盘页展示

复盘页可以完整展示：

```text
- 每次 attempt 的答案
- 正确答案
- 当时的 question_snapshot
- 当时题目版本
- 是否正确
- 解析
- 修改历史
```

复盘页不能和新答题页共用同一个渲染状态，否则容易再次把历史答案带回新 attempt。

建议前端拆成：

```text
QuestionPracticeCard     # 答题卡片，不显示历史答案
QuestionResultCard       # 提交后结果卡片
QuestionReviewCard       # 历史复盘卡片
```

## 8. 用户题目状态更新规则

每次 attempt 提交后，更新 `user_question_state`。

### 8.1 首次作答

如果不存在 `user_question_state`：

```text
创建一条记录。
```

如果答对：

```text
status = correct
attempt_count = 1
correct_count = 1
wrong_count = 0
consecutive_correct_count = 1
consecutive_wrong_count = 0
last_result = correct
next_review_at = now + 1 day 或 null
mastery_level = 1
```

如果答错：

```text
status = wrong
attempt_count = 1
correct_count = 0
wrong_count = 1
consecutive_correct_count = 0
consecutive_wrong_count = 1
last_result = wrong
next_review_at = now + 10 minutes 或 now + 1 hour
mastery_level = 0
```

### 8.2 再次作答

答对：

```text
attempt_count += 1
correct_count += 1
consecutive_correct_count += 1
consecutive_wrong_count = 0
last_result = correct
```

状态变化建议：

```text
if consecutive_correct_count >= 3:
    status = mastered
    next_review_at = null
else:
    status = reviewing 或 correct
    next_review_at = 根据掌握程度延后
```

答错：

```text
attempt_count += 1
wrong_count += 1
consecutive_wrong_count += 1
consecutive_correct_count = 0
status = wrong
next_review_at = 短间隔复习时间
mastery_level = max(0, mastery_level - 1)
```

### 8.3 简单复习间隔策略

先实现一个简单可控版本，不必一开始就做复杂算法：

| 情况 | next_review_at |
|---|---|
| 首次答错 | 当前时间 + 10 分钟 |
| 连续答错 | 当前时间 + 30 分钟 |
| 答对 1 次 | 当前时间 + 1 天 |
| 连续答对 2 次 | 当前时间 + 3 天 |
| 连续答对 3 次 | 当前时间 + 7 天 |
| 连续答对 4 次 | mastered，暂不复习 |

后续可以升级为更复杂的间隔复习算法。

## 9. API 设计建议

以下接口名称可以根据现有项目风格调整。

### 9.1 创建练习批次

```http
POST /api/practice-sessions
```

请求：

```json
{
  "mode": "new_random",
  "module": "Agent 基础与架构",
  "batch_size": 20,
  "filters": {
    "difficulty": null,
    "type": null,
    "knowledge_points": []
  },
  "allow_answered": false,
  "shortage_strategy": "error"
}
```

响应成功：

```json
{
  "session_id": 123,
  "mode": "new_random",
  "requested_count": 20,
  "actual_count": 20,
  "questions": [
    {
      "question_id": 1,
      "order_index": 1,
      "user_state": {
        "status": "new",
        "attempt_count": 0,
        "last_result": null
      }
    }
  ]
}
```

响应题目不足：

```json
{
  "code": "NOT_ENOUGH_NEW_QUESTIONS",
  "message": "当前模块未答题数量不足",
  "requested_count": 20,
  "available_count": 8,
  "suggested_actions": [
    "use_available_only",
    "switch_to_wrong_review",
    "include_answered",
    "reset_progress"
  ]
}
```

### 9.2 获取练习批次题目

```http
GET /api/practice-sessions/{session_id}
```

要求：

```text
- 返回 session 固定题目列表。
- 不重新随机。
- 未提交 attempt 不返回历史答案。
- 可以返回 user_state 标签。
```

### 9.3 开始某题 attempt

```http
POST /api/practice-sessions/{session_id}/questions/{question_id}/attempts
```

后端要求：

```text
1. 如果当前 session + question 已有未提交 attempt，则返回该 attempt。
2. 如果没有，则创建新 attempt。
3. 新 attempt 的 selected_answer / answer_text 必须为空。
4. question_snapshot 使用当前 session_question 里的 question_version 对应内容。
```

### 9.4 提交答案

```http
POST /api/attempts/{attempt_id}/submit
```

请求：

```json
{
  "selected_answer": ["B"],
  "answer_text": null
}
```

后端流程：

```text
1. 校验 attempt 属于当前用户。
2. 校验 attempt.status 未提交。
3. 根据 question_snapshot 判分。
4. 写入 selected_answer / answer_text / is_correct / score。
5. attempt.status = submitted 或 graded。
```

```python
def update_user_question_state_after_submit(user_id, question_id, question_version, attempt, result):
    state = get_or_create_user_question_state(user_id, question_id)

    state.current_question_version = question_version
    state.attempt_count += 1
    state.last_attempt_id = attempt.id
    state.last_attempt_at = attempt.submitted_at
    state.last_answer_snapshot = attempt.selected_answer or attempt.answer_text

    if result.is_correct:
        state.correct_count += 1
        state.consecutive_correct_count += 1
        state.consecutive_wrong_count = 0
        state.last_result = "correct"

        if state.consecutive_correct_count >= 4:
            state.status = "mastered"
            state.next_review_at = None
            state.mastery_level = 5
        else:
            state.status = "correct"
            state.mastery_level = min(5, state.mastery_level + 1)
            state.next_review_at = calculate_next_review_at(state)
    else:
        state.wrong_count += 1
        state.consecutive_wrong_count += 1
        state.consecutive_correct_count = 0
        state.last_result = "wrong"
        state.status = "wrong"
        state.mastery_level = max(0, state.mastery_level - 1)
        state.next_review_at = calculate_next_review_at(state)

    save(state)
```

## 13. 抽题伪代码

```python
def create_practice_session(
    user_id,
    mode,
    module=None,
    batch_size=20,
    filters=None,
    allow_answered=False,
    shortage_strategy="error",
):
    with transaction():
        candidates = select_candidate_questions(
            user_id=user_id,
            mode=mode,
            module=module,
            filters=filters,
            allow_answered=allow_answered,
        )

        available_count = len(candidates)

        if available_count < batch_size:
            if shortage_strategy == "error":
                return not_enough_questions_response(
                    requested_count=batch_size,
                    available_count=available_count,
                )
            elif shortage_strategy == "use_available_only":
                selected = candidates
            elif shortage_strategy == "include_answered":
                selected = fill_with_answered_questions(candidates, batch_size)
            else:
                raise ValueError("Unsupported shortage strategy")
        else:
            selected = candidates[:batch_size]

        session = PracticeSession(
            user_id=user_id,
            mode=mode,
            module=module,
            requested_count=batch_size,
            actual_count=len(selected),
            status="active",
            allow_answered=allow_answered,
            show_history_policy=get_show_history_policy(mode),
        )
        save(session)

        for index, question in enumerate(selected, start=1):
            save(PracticeSessionQuestion(
                practice_session_id=session.id,
                question_id=question.id,
                question_version=question.version,
                order_index=index,
                source_reason=get_source_reason(mode, question),
            ))

        return session
```

## 19. 最终实现原则

本次改造的最终原则如下：

```text
1. 抽题不是全题库 random，而是从当前用户的合法候选池中选择。
2. 新题模式默认排除已答题。
3. 已答题只能进入复习、全量练习、复盘等明确模式。
4. 练习批次一旦生成，必须固定，不随刷新变化。
5. 历史答案永远不能预填到新 attempt。
6. 答题页和复盘页必须分离。
7. 题目不足时必须提示，不得静默重复。
8. Attempt 记录一次行为，UserQuestionState 记录累计状态，PracticeSession 记录一批题。
```
