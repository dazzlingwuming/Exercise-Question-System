# 手动新增题目功能设计

## 一、背景

当前系统已经支持：

* Markdown 题库导入
* 题目编辑
* 题目软删除 / 恢复
* 修改历史
* Practice Session
* 错题复习
* 统计

但目前新增题目的主要入口仍然是：

```text
编写 Markdown 题库文件
        ↓
解析预览
        ↓
导入数据库
```

这对批量题库很合适，但不适合用户在使用过程中临时补充单题。

现在需要新增：

```text
用户可以在页面上手动新增一道题
```

并且支持不同题型、难度、方向、考察点、解析、评分标准等属性。

## 二、核心目标

新增一个页面：

```text
/questions/new
```

名称：

```text
新增题目
```

用户可以通过表单创建题目。保存后，题目进入正式题库，并默认参与：

* 题库列表
* 随机练习
* 按导入顺序练习
* 按方向练习
* 按考察点练习
* 未答题练习
* 统计

如果用户后续发现题目有误，可以继续使用现有编辑、历史、删除、恢复能力。

## 三、产品原则

### 1. 手动新增不是临时草稿

第一版不做复杂草稿系统。

点击保存成功后：

```text
直接写入 questions 表
```

题目立即成为正式题目。

### 2. 新增题目也必须结构化

不能只保存一段纯文本。

至少要结构化保存：

```text
题型
难度
题干
标准答案 / 参考答案
解析
考察点
方向
```

选择题还必须保存：

```text
选项 A/B/C/D...
```

### 3. 新增题目也必须可追溯

虽然题目是新建的，不是修改旧题，但仍建议生成一条 revision 或 creation log。

推荐复用现有 `question_revisions` 表，创建时写入：

```text
source = "manual_create"
changed_fields = ["created"]
reason = 用户填写的创建说明
before_data = {}
after_data = 新题目快照
version_before = 0
version_after = 1
```

这样题目的生命周期完整：

```text
创建 -> 编辑 -> 删除 -> 恢复
```

都有历史记录。

## 四、支持的题型

第一版建议支持现有系统内部题型：

| 题型 | type | 判分方式 |
|---|---|---|
| 单选题 | single_choice | 自动判分 |
| 多选题 | multiple_choice | 自动判分 |
| 判断题 | true_false | 自动判分 |
| 填空题 | fill_blank | 自动判分 |
| 简答题 | short_answer | 用户自评 |
| 概念辨析题 | concept_analysis | 用户自评 |
| 场景分析题 | scenario_analysis | 用户自评 |
| 面试题 | interview | 用户自评 |
| Debug / 日志分析题 | debug_analysis | 用户自评 |
| 代码阅读 / 伪代码设计题 | code_reading | 用户自评 |
| 系统设计题 | system_design | 用户自评 |
| 项目追问题 | project_follow_up | 用户自评 |
| 模拟面试套卷 | mock_interview | 用户自评 |

不建议让用户输入任意未知题型。

原因：

* 未知题型无法稳定渲染
* 无法确定判分方式
* 练习页和编辑页需要额外兼容

如果后续确实需要扩展题型，应先在系统题型枚举和前端渲染器中注册。

## 五、字段设计

### 1. 基础字段

新增题目表单建议包含：

```text
题型 type
题型显示名 type_label
难度 difficulty
标签 tags
方向 directions
考察点 exam_points
题干 stem
材料 material
选项 options
标准答案 standard_answer
详细解析 explanation
常见错误 common_mistakes
面试延伸追问 follow_up_question
评分标准 scoring_standard
创建说明 reason
```

### 2. 自动生成字段

这些字段不要求用户手动填写：

```text
id
part_id
title
import_order
source_text
parse_warnings
version
is_deleted
created_at
updated_at
```

### 3. id 生成规则

建议后端生成稳定 id：

```text
manual_{uuid}
```

例如：

```text
manual_7f5a3d0c2a6b4f2e9f0b1c8d9e123456
```

不要让前端自己生成主键。

### 4. part_id 生成规则

建议后端生成：

```text
Manual-{递增序号}
```

例如：

```text
Manual-000061
```

生成方式：

```text
当前 questions 表最大 import_order + 1
```

然后：

```text
part_id = Manual-{import_order:06d}
import_order = 最大 import_order + 1
```

这样手动新增题目会自然排在按导入顺序练习的后面。

### 5. title 生成规则

建议：

```text
title = "{type_label}｜{difficulty or 未标注}"
```

或者：

```text
title = stem 前 40 个字符
```

第一版可以用：

```text
title = part_id
```

保持简单。

### 6. source_text 生成规则

手动新增没有原始 Markdown，可以由后端生成一段可读 source：

```text
source_text = "manual_create\n\n题干：...\n\n标准答案：..."
```

这样旧的详情、历史、快照逻辑不会因为 source_text 为空而出问题。

## 六、不同题型的表单规则

### 1. 单选题

必填：

```text
题干
至少 2 个选项
标准答案
```

标准答案必须是某一个已有选项 key：

```text
A
B
C
```

前端交互：

```text
用 radio 选择正确答案
```

### 2. 多选题

必填：

```text
题干
至少 2 个选项
标准答案
```

标准答案必须是多个已有选项 key 的集合。

前端交互：

```text
用 checkbox 选择多个正确答案
```

保存时可以归一化为：

```text
["A", "C"]
```

或者：

```text
"A、C"
```

建议后端统一保存为当前系统已经兼容的格式，优先不改判分器。

### 3. 判断题

必填：

```text
题干
标准答案
```

前端固定两个选项：

```text
A. 正确
B. 错误
```

标准答案可保存为：

```text
正确
```

或：

```text
A
```

建议使用现有判断题判分器已经支持的表达：

```text
正确 / 错误
```

### 4. 填空题

必填：

```text
题干
标准答案
```

标准答案可以允许多个可接受答案：

```text
答案1、答案2、答案3
```

第一版可以先保存为字符串。

后续如果要更严格，可以改成数组。

### 5. 主观题

包括：

```text
简答题
概念辨析题
场景分析题
面试题
Debug / 日志分析题
代码阅读 / 伪代码设计题
系统设计题
项目追问题
模拟面试套卷
```

必填：

```text
题干
参考答案
```

强烈建议填写：

```text
评分标准
考察点
```

这些题型提交后走用户自评，不自动判分。

## 七、后端 API 设计

### 1. 新增题目

新增接口：

```http
POST /api/questions
```

请求体：

```json
{
  "type": "single_choice",
  "type_label": "单选题",
  "difficulty": "2",
  "tags": ["手动新增"],
  "directions": ["Tool Calling"],
  "stem": "下面哪项说法正确？",
  "material": null,
  "options": [
    { "key": "A", "text": "错误选项" },
    { "key": "B", "text": "正确选项" }
  ],
  "standard_answer": "B",
  "explanation": "B 正确。",
  "exam_points": ["工具调用", "参数校验"],
  "common_mistakes": "把 schema 描述当成强校验。",
  "follow_up_question": "如果 strict mode 不可用，你怎么兜底？",
  "scoring_standard": null,
  "reason": "手动补充工具调用基础题"
}
```

返回：

```json
{
  "id": "manual_xxx",
  "part_id": "Manual-000061",
  "type": "single_choice",
  "type_label": "单选题",
  "version": 1,
  "is_deleted": false
}
```

### 2. API 行为

后端执行：

1. 校验题型是否合法
2. 校验题干不为空
3. 校验客观题答案和选项匹配
4. 校验主观题至少有参考答案或评分标准
5. 生成 `id`
6. 生成 `part_id`
7. 生成 `import_order`
8. 生成 `source_text`
9. 写入 `questions`
10. 写入 `question_revisions`
11. 返回 `QuestionRead`

### 3. 错误返回

示例：

```json
{
  "detail": "单选题标准答案必须是一个已有选项"
}
```

```json
{
  "detail": "题干不能为空"
}
```

```json
{
  "detail": "主观题至少需要参考答案或评分标准"
}
```

## 八、后端服务设计

建议新增：

```text
backend/app/services/question_create_service.py
```

核心函数：

```python
def create_question(db: Session, payload: QuestionCreate) -> Question:
    """
    中文说明：
    手动新增一道题目。新增题目直接写入正式 questions 表，
    自动生成 id、part_id、import_order 和 source_text，
    并生成 manual_create 修改历史，保证题目生命周期可追溯。
    """
```

辅助函数：

```python
def next_import_order(db: Session) -> int
```

```python
def build_manual_part_id(import_order: int) -> str
```

```python
def build_manual_source_text(payload: QuestionCreate) -> str
```

```python
def build_creation_revision(question: Question, reason: str | None) -> QuestionRevision
```

## 九、Schema 设计

新增：

```python
class QuestionCreate(BaseModel):
    type: str
    type_label: str | None = None
    difficulty: str | None = None
    tags: list[str] = []
    directions: list[str] = []
    stem: str
    material: str | None = None
    options: list[OptionSchema] = []
    standard_answer: Any
    explanation: str | None = None
    exam_points: list[str] = []
    common_mistakes: str | None = None
    follow_up_question: str | None = None
    scoring_standard: str | None = None
    reason: str | None = None
```

注意：

```text
QuestionCreate 不允许传 id、part_id、import_order、version、is_deleted
```

这些必须由后端生成。

## 十、前端页面设计

### 1. 入口

在题库页顶部增加：

```text
[新增题目]
```

跳转：

```text
/questions/new
```

侧边栏是否新增入口可选。

第一版建议只在题库页顶部加入口，避免侧边栏过多。

### 2. 页面布局

页面标题：

```text
新增题目
```

页面结构建议复用编辑页：

```text
基础信息
题干内容
选项设置
标准答案 / 参考答案
解析与考察点
创建说明
保存按钮
```

### 3. 表单交互

选择题型后动态变化：

* 单选 / 多选：显示选项编辑器和选项答案选择器
* 判断题：显示正确 / 错误选择器，不显示手动选项编辑
* 填空题：显示文本答案输入
* 主观题：显示参考答案大文本框和评分标准

### 4. 保存成功后

保存成功后跳转：

```text
/questions/{id}
```

并显示：

```text
题目创建成功
```

### 5. 取消

取消返回：

```text
/questions
```

如果表单已填写内容，可以提示：

```text
当前内容未保存，确认离开？
```

第一版可以先不做复杂离开保护。

## 十二、与现有功能的关系

### 1. 与题库列表

新增题目保存后：

```text
GET /api/questions
```

默认可以查到。

### 2. 与练习

新增题目默认：

```text
is_deleted = false
```

因此会进入新建 Practice Session。

按导入顺序练习时，它排在最后。

### 3. 与未答题练习

新题没有 attempt。

因此会进入：

```text
mode=unanswered
```

### 4. 与错题

新题不会进入错题，除非用户答错。

### 5. 与统计

新题会增加：

```text
total_questions
```

未答题数量也会相应增加。

### 6. 与删除

新题可以走现有软删除。

### 7. 与修改历史

创建时生成 `manual_create` revision。

后续编辑继续生成 `manual_edit` revision。

## 十三、权限和安全

本地系统当前没有多用户和登录。

所以第一版无需做用户权限。

但仍要防止：

* 空题干
* 错误题型
* 客观题答案不存在
* 空标准答案
* 选项 key 重复
* 过长字段导致页面失控

建议后端限制：

```text
stem <= 20000 字符
explanation <= 50000 字符
options <= 26 个
option.text <= 5000 字符
```

前端也做基本提示，但以后端校验为准。

## 十四、测试要求

### 后端 pytest

至少覆盖：

1. 创建单选题成功
2. 创建多选题成功
3. 创建判断题成功
4. 创建主观题成功
5. 单选题答案不在选项中时失败
6. 多选题答案包含不存在选项时失败
7. 空题干失败
8. 主观题没有参考答案和评分标准时失败
9. 创建后 `version = 1`
10. 创建后 `is_deleted = false`
11. 创建后 `import_order` 是当前最大值 + 1
12. 创建后生成 `manual_create` revision
13. 创建后题库列表可查到
14. 创建后未答题练习可查到
15. 创建后 Practice Session 可包含该题

### 前端

至少保证：

```bash
npm run build
```

通过。

## 十六、第一版不做的事情

为了避免范围过大，第一版不建议做：

* 草稿箱
* 批量手动新增
* 从页面导出 Markdown
* AI 自动生成题目
* 多用户创建人字段
* 富文本编辑器
* 图片题
* 附件题
* 题目合并

这些可以后续扩展。

## 十七、结论

推荐实现方式：

```text
新增 POST /api/questions
        ↓
后端统一生成 id / part_id / import_order
        ↓
复用现有题目校验逻辑
        ↓
写入 questions
        ↓
生成 manual_create revision
        ↓
前端新增 /questions/new 页面
        ↓
保存后跳转题目详情
```

这个方案和现有系统兼容度最高，不需要新增数据库表，也不会破坏导入、练习、错题、统计和软删除逻辑。
