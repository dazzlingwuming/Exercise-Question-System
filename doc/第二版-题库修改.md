## 三、重要设计原则

### 1. 不要直接无痕覆盖

题目修改不能只是简单覆盖数据库字段。

必须保留修改历史，至少记录：

* 修改前的数据
* 修改后的数据
* 修改时间
* 修改字段
* 修改原因，可选
* 修改来源，例如 manual_edit

这样后续如果改错了，可以回看或恢复。

### 2. 后续判分使用新答案

用户修改标准答案后：

* 新的刷题提交必须按新标准答案判分
* 旧的答题记录不要强行重算
* 旧记录可以保留原来的判分结果
* 如果需要，在题目详情里提示“该题已被修改，历史答题记录可能基于旧版本”

### 3. 尽量记录题目版本

建议给 Question 增加：

```python
version: int
updated_at: datetime
```

每修改一次题目，version + 1。

Attempt 答题记录中建议增加：

```python
question_version: int | None
question_snapshot: dict | None
```

这样后续可以知道用户当时是基于哪个版本作答。

如果已有历史 attempt 没有 version，可以保持为空，不要破坏旧数据。

### 4. 编辑必须做校验

保存题目前必须校验：

* 题干不能为空
* 客观题标准答案不能为空
* 单选题标准答案必须是一个选项
* 多选题标准答案必须是已有选项集合
* 判断题标准答案必须能归一化为正确/错误
* 填空题标准答案不能为空
* 主观题可以没有自动判分答案，但最好有参考答案或评分标准
* 选项 key 不能重复
* 选项文本不能为空
* 题型变更后，答案格式必须符合新题型

不要让用户保存明显无法判分的数据。

## 四、后端需要新增或修改的内容

### 1. 数据库新增 question_revisions 表

请新增一个题目修改历史表。

建议模型：

```python
QuestionRevision:
    id: str
    question_id: str
    version_before: int
    version_after: int

    before_data: dict
    after_data: dict
    changed_fields: list[str]

    reason: str | None
    source: str
    created_at: datetime
```

字段说明：

* `before_data`：修改前完整题目数据
* `after_data`：修改后完整题目数据
* `changed_fields`：本次修改了哪些字段
* `reason`：用户填写的修改原因
* `source`：默认 `manual_edit`
* `created_at`：修改时间

如果 SQLite JSON 字段处理不方便，可以用 Text 存 JSON 字符串，但必须封装好序列化和反序列化逻辑，并写中文注释说明。

### 2. Question 模型新增字段

给 questions 表增加：

```python
version: int = 1
updated_at: datetime
```

如果当前表里没有这些字段，需要写一个安全的轻量迁移逻辑，启动后端时检查字段是否存在，缺失则补充。

不要要求我手动删库重建。

### 3. Attempt 模型建议新增字段

给 attempts 表增加：

```python
question_version: int | None
question_snapshot: str | None
```

提交答案时，把当时的题目版本和核心题目信息快照保存进去。

这样后续即使题目被修改，历史答题记录也能追踪当时作答的题目内容。

### 4. 新增更新题目 API

新增接口：

```http
PATCH /api/questions/{question_id}
```

请求体示例：

```json
{
  "stem": "新的题干",
  "options": [
    {"key": "A", "text": "选项 A"},
    {"key": "B", "text": "选项 B"}
  ],
  "standard_answer": "B",
  "explanation": "新的解析",
  "exam_points": ["工作流 Agent", "可观测性"],
  "common_mistakes": "常见错误说明",
  "follow_up_question": "面试追问",
  "scoring_standard": "评分标准",
  "difficulty": "进阶",
  "tags": ["高仿真题"],
  "reason": "原标准答案有误，修正为 B"
}
```

要求：

* 支持部分字段更新
* 只更新传入字段
* 保存前做字段校验
* 保存修改历史
* question.version + 1
* 返回更新后的题目

### 5. 新增题目修改历史 API

新增接口：

```http
GET /api/questions/{question_id}/revisions
```

返回该题所有修改历史，按时间倒序。

返回内容包括：

* revision id
* version_before
* version_after
* changed_fields
* reason
* source
* created_at

详情中可以包含 before_data / after_data，但列表可以只返回摘要，避免太重。

### 6. 新增修改历史详情 API

```http
GET /api/questions/{question_id}/revisions/{revision_id}
```

返回完整修改前后数据，方便前端展示对比。

### 7. 新增恢复历史版本 API

```http
POST /api/questions/{question_id}/revisions/{revision_id}/restore
```

作用：

* 将题目恢复到某个 revision 的 before_data 或 after_data
* 建议默认恢复到 before_data
* 恢复本身也要产生一条新的 revision
* version 继续 +1，不要倒退版本号

请求体可以支持：

```json
{
  "restore_target": "before",
  "reason": "上一次修改有误，恢复旧版本"
}
```
