stem: |
  设计一个多智能体的旅行规划系统。用户提出行程需求后，系统的旅行智能体将分别并行调用天气智能体和景点智能体等多个智能体，最终综合信息给出完整行程规划。请描述该系统的核心组件与数据流，以及如何评估其性能和潜在风险。
reference_answer: |
  **核心组件**：旅行智能体（Travel Agent）负责整体交互和结果整合；天气智能体（Weather Agent）查询目的地天气信息；旅游景点智能体（Attraction Agent）检索景点信息；如果系统基于Ray A2A架构，每个智能体可以作为独立服务注册Agent Card。  
  **数据流**：用户请求进入旅行智能体。旅行智能体并行调用天气智能体和景点智能体（可通过A2A协议或并行工具调用），同时可能调用其他服务（如交通Agent）。各智能体返回数据给旅行智能体，旅行智能体负责综合多源信息并生成最终行程。结果返回用户。  
  **评估指标**：设计行程的准确度（是否满足用户需求）、响应时延、用户满意度、系统并发吞吐量等。可以量化天气预测准确率或景点推荐相关性等子指标。  
  **风险点**：各智能体数据一致性风险，例如天气和景点信息时间不同步；网络请求失败导致部分信息缺失；A2A调用延迟；错误合成导致不合理行程；隐私安全问题（处理用户偏好和位置信息时的保护）。需要设计超时与重试机制、验证流程等。
architecture_points:
  - 旅行智能体：作为协同Orchestrator，分发任务并汇总行程建议。
  - 天气智能体：并行查询天气API，根据日期和地点提供天气信息。
  - 景点智能体：检索旅游信息库或web搜索，返回推荐景点和活动列表。
  - (可选) 交通智能体：提供交通和交通时间信息。
data_flow:
  - 用户输入旅行需求（时间、地点、偏好）→旅行Agent接收。
  - 旅行Agent并行调用天气Agent和景点Agent获取数据（并行工具调用或A2A消息）。
  - 各子Agent返回信息后，旅行Agent对信息进行整合（如生成每日行程、交通安排）。
  - 最终生成行程计划并返回用户。
evaluation_metrics:
  - 计划质量：通过用户反馈或自动评估指标（如覆盖主要景点，行程可行性）衡量。
  - 响应时间：完成规划所需的总时间，包括并行调用延迟。
  - 系统吞吐量：单位时间可处理的请求数量。
  - 成功率：系统能完整返回合格行程的比例。
risk_points:
  - 数据一致性：各Agent数据来源不同，可能存在时效或冲突，需同步机制。
  - 调用延迟或失败：并行调用中某Agent响应慢或失败需超时和重试策略。
  - 结果整合错误：多个源信息合成时可能出现逻辑冲突，如冲突日期安排。
  - 用户隐私：处理位置和个人偏好时需保护隐私安全，避免敏感信息泄露。
interview_followups:
  - 如果天气Agent返回空数据，旅行Agent应如何处理？
  - 在并行调用中，如果一个子Agent异常慢或挂掉，如何保证系统仍能响应？

type: system_design
difficulty: 5
knowledge_points:
  - 分层架构
  - 评估指标
  - 安全风险
stem: |
  设计一个多智能体软件开发系统。该系统包含一个规划（Planner）智能体，用于将复杂需求分解为子任务并协调执行，以及多个专门执行子任务的智能体（如编码、测试、文档）。请描述系统的核心架构和组件，说明数据流动过程，以及评估该系统成功的指标和潜在风险。
reference_answer: |
  **核心组件**：规划智能体（Planner）处于战略层，接受开发需求并分解任务；多个执行智能体（Coder Agent、Tester Agent、Doc Agent等）位于战术/执行层，分别负责编码、编写测试、生成文档等。可增设验证智能体（Verifier）检查各阶段成果。系统采用层级架构，Planner协调下级Agent。  
  **数据流动**：用户需求输入Planner，Planner生成任务列表并分配给相应执行Agent。执行Agent运行工具/LLM完成任务，将结果返回给Planner或Verifier。验证Agent对输出进行质量检查，通过结果写入共享规范（Living Spec）。若有不合格，Planner调整任务分配继续执行。最终结果整合交给用户。  
  **评估指标**：功能实现正确率、交付时间、质量（如测试通过率、文档完整度）、自动化覆盖率。可以统计各Agent的吞吐、错误率、任务完成度来评价系统性能。  
  **风险点**：任务分解可能不合理导致遗漏或重复；Agent间接口协议不一致造成通信失败；上下文漂移（长对话中早期信息丢失）；循环依赖或无限迭代；安全边界（不同Agent访问代码库权限）。需设计严格的输入输出合同（schema）、监控Agent状态和终止条件。
architecture_points:
  - 规划智能体（Strategic Planner）：根节点Agent，负责目标理解、任务分解与调度。
  - 执行智能体（Operational Agents）：如编码Agent、测试Agent、文档Agent等，专注具体子任务。
  - 验证智能体（Verifier）：独立检查Agent输出结果质量，确保符合需求和规范。
  - 协调层级：规划Agent和执行Agent形成层次结构，信息通过共享状态或工具调用交换。
data_flow:
  - 用户需求输入→Planner解析并拆分子任务→调用对应子Agent执行→子Agent输出返回给Planner/Verifier→Verifier检查并反馈结果→迭代至完成→最终结果输出给用户。
  - 流程中可采用Living Spec等形式将规范写入共享存储，供各Agent参考和更新。
evaluation_metrics:
  - 功能覆盖率：实现需求点的完整度。
  - 开发效率：总开发时间、迭代次数。
  - 质量指标：测试通过率、代码错误率、文档质量评分等。
  - 自动化程度：人工干预次数、Agent自主解决问题的比例。
risk_points:
  - 分解偏差：Planner拆分任务可能遗漏关键步骤或分配错误，需人工审查。
  - 上下文丢失：长时间协作中，早期决策信息可能遗忘或被覆盖，需引入持续规范或摘要机制。
  - 无限循环：验证失败导致Planner反复调度任务，可能出现迭代循环，需要退出条件。
  - 权限与安全：不同Agent可能需要访问代码库或测试环境，需要安全边界和审计机制。
interview_followups:
  - 如果某个子任务需要等待外部事件（如代码审查通过），如何在系统中处理这种异步逻辑？
  - 在系统负载较高时，如何动态扩展Agent实例或调整任务分配策略？

stem: |
  编写伪代码实现一个简单的智能体交接（Handoff）机制：当前智能体根据任务状态决定是否将任务交给下一个智能体。假设我们有两个智能体A和B，当状态为“needs_B”时交接给B，否则A继续处理。请提供核心逻辑的伪代码，并说明时间和空间复杂度。
reference_answer: |
  核心思路是根据任务状态判断交接条件，如果满足则调用另一个智能体，否则由当前智能体继续执行。伪代码可以使用函数或字典返回下一个操作。
pseudocode: |
    def agent_node(state):
        # state 包含任务信息和当前状态
        if state.get("next_agent") == "B":
            # 设置要传递给B的状态信息
            next_state = state.copy()
            next_state["handled_by"] = "B"
            return {"handoff": "B", "state": next_state}
        else:
            # 当前智能体A继续处理
            result = process_by_A(state)
            # 根据处理结果决定是否需要交接
            if result.indicates_handoff():
                next_state = state.copy()
                next_state["next_agent"] = "B"
                return {"handoff": "B", "state": next_state}
            else:
                # 完成任务返回结果
                return {"result": result}
complexity:
  time: O(1)  # 判断和调用操作是常数时间
  space: O(n) # 复制状态所需空间，n为状态大小
test_cases:
  - input:
      state: { "task": "calculate", "next_agent": "B" }
    expected:
      handoff: "B"
  - input:
      state: { "task": "calculate", "next_agent": "A" }
    expected:
      result: "<A完成后的结果>"
answer_keywords:
  - 状态判断
  - 交接工具
  - 状态复制
scoring_points:
  - 检查伪代码是否正确判断状态并执行交接。
  - 包含了状态复制和设置下一智能体标记的步骤。
  - 时间复杂度和空间复杂度分析的说明正确。

stem: |
  编写伪代码实现一个简单的任务调度器，将任务列表分发给多个智能体并收集结果。假设输入为一个任务列表和一个智能体列表，每个智能体处理一个任务即可。请提供核心逻辑的伪代码，并说明时间和空间复杂度。
reference_answer: |
  思路：遍历任务列表，为每个任务选择一个智能体（例如轮询或随机分配），调用该智能体执行任务并收集结果。可以将结果存储在列表中返回。  
pseudocode: |
    def schedule_tasks(tasks, agents):
        results = []
        for i, task in enumerate(tasks):
            agent = agents[i % len(agents)]  # 简单轮询分配
            result = agent.process(task)
            results.append(result)
        return results
complexity:
  time: O(m)  # m为任务数量，因为为每个任务调用一次处理
  space: O(m)  # 存储结果列表需要与任务数量成正比的空间
test_cases:
  - input:
      tasks: [1, 2, 3]
      agents: [AgentA, AgentB]
    expected:
      results: [AgentA.process(1), AgentB.process(2), AgentA.process(3)]
  - input:
      tasks: []
      agents: [AgentA]
    expected:
      results: []
answer_keywords:
  - 任务分发
  - 循环调度
  - 并行执行
scoring_points:
  - 伪代码能正确遍历任务并分配给智能体执行。
  - 使用合理的分配策略（如轮询）并收集返回结果。
  - 对时间和空间复杂度给出了正确的分析。
