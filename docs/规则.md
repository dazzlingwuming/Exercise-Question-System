## 四、选择题标准格式

这是你最应该固定下来的格式。

```md
--- question ---

question_id: agent_basic_sc_001
module: Agent 基础与架构
type: single_choice
difficulty: 3
knowledge_points:
  - Agent Loop
  - ReAct Agent
  - 状态管理

stem: |
  关于 Agent 和普通 LLM Chatbot 的区别，下列说法最准确的是？

options:
  A: Agent 本质上只是更长的 Prompt
  B: Agent 通常具备目标驱动、多步决策、工具调用和状态更新能力
  C: 普通 Chatbot 一定不能调用工具
  D: Agent 不需要应用层控制，模型可以自动完成所有执行

answer:
  - B

option_analysis:
  A: 错误。Agent 不是简单加长 Prompt，而是围绕任务目标进行多步决策和执行。
  B: 正确。Agent 通常包含规划、工具调用、状态更新、结果观察和继续决策等环节。
  C: 错误。普通 Chatbot 也可以接入工具，但不一定具备完整的任务执行循环。
  D: 错误。Agent 仍然需要应用层负责工具执行、权限控制、状态管理和异常处理。

interview_followups:
  - Agent 和 Workflow 的区别是什么？
  - 什么场景下不应该使用 Agent？
  - Agent Loop 中每一步分别承担什么作用？
```

你的选择题字段固定为：

```text
question_id
module
type
difficulty
knowledge_points
stem
options
answer
option_analysis
interview_followups
```

其中 `answer` 必须永远是数组：

```yaml
answer:
  - B
```

多选题就是：

```yaml
answer:
  - A
  - C
  - D
```

这样后端判断会稳定很多。

## 七、概念辨析题格式

概念辨析题不需要 `options`，而是用 `reference_answer`。

```md
--- question ---

question_id: agent_basic_cc_001
module: Agent 基础与架构
type: concept_compare
difficulty: 4
knowledge_points:
  - Agent
  - Workflow
  - 系统边界

stem: |
  请辨析 Agent 和 Workflow 的区别。什么情况下应该使用 Agent，什么情况下应该使用 Workflow？

reference_answer: |
  Agent 更适合目标明确但路径不完全固定、需要模型动态决策、选择工具和处理不确定性的任务。Workflow 更适合流程稳定、步骤明确、可预测性要求高的任务。
  
  两者的核心区别不在于是否调用大模型，而在于控制权在哪里。Workflow 通常由应用层预先定义执行路径；Agent 则允许模型在一定约束下动态决定下一步动作。
  
  如果任务需要强确定性、审计和稳定执行，例如固定审批流、标准 ETL、定时任务，优先使用 Workflow。如果任务需要根据上下文动态选择工具、处理开放问题、跨步骤推理，可以考虑 Agent。

answer_keywords:
  - 动态决策
  - 固定流程
  - 工具选择
  - 控制权
  - 确定性
  - 不确定性

scoring_points:
  - 能说明 Agent 是动态决策，不只是调用大模型
  - 能说明 Workflow 是预定义流程
  - 能结合使用场景做取舍
  - 能指出 Agent 成本、延迟和稳定性风险

interview_followups:
  - 为什么不能所有任务都做成 Agent？
  - Agent 和 Workflow 能不能混合使用？
  - 如果业务流程固定，但中间需要模型判断，应该怎么设计？
```

简答类题目建议字段：

```text
question_id
module
type
difficulty
knowledge_points
stem
reference_answer
answer_keywords
scoring_points
interview_followups
```

## 八、场景排障题格式

这是你面试最应该重点训练的题型。

```md
--- question ---

question_id: rag_sd_001
module: RAG / Agentic RAG / 向量检索
type: scenario_debug
difficulty: 4
knowledge_points:
  - RAG 排障
  - 检索召回
  - 上下文注入
  - 幻觉问题

stem: |
  一个企业知识库 RAG 系统中，用户提问后，系统检索到了包含正确答案的文档片段，但模型最终回答仍然出现幻觉。你会如何排查和优化？

reference_answer: |
  需要从检索、排序、上下文构造、Prompt、模型输出和评估链路逐步排查。
  
  首先确认正确片段是否真的进入了最终 Prompt，而不是只在检索结果中出现。其次检查正确片段的位置是否过靠后，是否被无关片段稀释。然后检查 Prompt 是否要求模型基于证据回答，是否允许不知道时拒答。还需要检查 chunk 切分是否破坏语义，rerank 是否把关键片段排低，引用和答案生成是否绑定。
  
  优化上可以使用 rerank、上下文压缩、证据优先排序、引用约束、答案一致性校验、低置信度拒答，以及构建离线评测集观察召回率、命中率和答案正确率。

diagnosis_steps:
  - 检查正确文档是否进入最终上下文
  - 检查正确片段在上下文中的排序位置
  - 检查无关片段是否过多
  - 检查 Prompt 是否要求基于证据回答
  - 检查 chunk 切分是否破坏语义
  - 检查 rerank 是否有效
  - 检查模型是否忽略证据自行生成

optimization_points:
  - rerank
  - context compression
  - evidence-first ordering
  - citation grounding
  - answer verification
  - low-confidence refusal
  - offline evaluation

interview_followups:
  - 如何区分是检索问题还是生成问题？
  - RAG 评估应该看哪些指标？
  - 如果召回正确但答案仍错，是否应该换更大的模型？
```

场景题字段建议：

```text
reference_answer
diagnosis_steps
optimization_points
interview_followups
```

这类题比普通选择题更接近真实面试。

## 十、代码 / 伪代码题格式

代码题最好不要直接要求完整代码，可以先支持伪代码。

```md
--- question ---

question_id: algorithm_code_001
module: 算法与数据结构 / AI 应用算法
type: coding_or_pseudocode
difficulty: 3
knowledge_points:
  - TopK
  - 堆
  - 向量检索

stem: |
  给定一组文档向量和一个查询向量，请设计一个 TopK 相似文档检索函数，返回相似度最高的 K 个文档。可以使用余弦相似度。

reference_answer: |
  核心思路是遍历所有文档向量，计算每个文档与查询向量的余弦相似度，然后维护一个大小为 K 的最小堆。堆中保存当前相似度最高的 K 个文档。当新文档分数高于堆顶时，弹出堆顶并插入新文档。最终堆中元素就是 TopK 结果。
  
  如果数据规模较小，可以直接全量排序，复杂度为 O(n log n)。如果数据较大，使用大小为 K 的堆，复杂度为 O(n log k)。生产环境中通常会使用向量数据库或 ANN 索引，而不是每次全量扫描。

pseudocode: |
  function top_k_search(query_vector, documents, k):
      heap = empty_min_heap()
      for doc in documents:
          score = cosine_similarity(query_vector, doc.vector)
          if heap.size < k:
              heap.push((score, doc.id))
          else if score > heap.peek().score:
              heap.pop()
              heap.push((score, doc.id))
      return heap.sorted_desc()

complexity:
  time: O(n log k)
  space: O(k)

test_cases:
  - input: 查询向量与某个文档完全相同
    expected: 该文档应该排在第一位
  - input: k 大于文档数量
    expected: 返回所有文档并按相似度排序
  - input: 存在多个相同相似度
    expected: 可以按文档 ID 或原始顺序稳定排序

interview_followups:
  - 为什么不用全量排序？
  - 余弦相似度和点积有什么区别？
  - 如果文档量达到千万级，如何优化？
```

代码题字段建议：

```text

pseudocode
complexity
test_cases
interview_followups
```

## 题目要求
题目质量硬性要求：

1. 不得高度重复

* 同一模块内不得出现语义高度重复的题目。
* 同一知识点可以出现多道题，但必须从不同角度考察，例如概念理解、工程场景、排障路径、系统设计、代码实现。
* 不允许只是替换措辞、题干顺序或同义词来生成新题。

2. 题干必须明确

* 题干必须明确考察对象、场景边界和问题重点。
* 不允许出现过于宽泛、无法判断答案的问题。
* 不允许出现题干信息不足，导致多个答案都合理的情况。

3. 选择题答案必须唯一且一致

* single_choice 必须只有一个明显最优答案。
* multiple_choice 必须有两个或以上正确答案。
* true_false 必须只有“正确 / 错误”两个选项。
* answer 必须是数组格式，不能写成自然语言。
* answer 中的选项必须存在于 options。
* 不允许出现 answer 和 options 不匹配的问题。

4. 选项质量要求

* 选项之间不能重复或语义高度相似。
* 干扰项必须来自真实常见误区，不能使用明显荒谬、无意义、凑数的选项。
* 单选题不能出现多个都正确的选项。
* 多选题不能出现正确选项数量不符合答案的问题。

5. 选项解析必须与答案一致

* option_analysis 必须覆盖所有选项。
* 正确答案对应的 option_analysis 必须明确说明“正确”及原因。
* 错误答案对应的 option_analysis 必须明确说明“错误”及原因。
* option_analysis 不得与 answer 字段冲突。
* 解析不能只是重复选项文本，必须解释背后的原理、工程原因或常见误区。

6. 主观题答案必须结构化

* concept_compare 题必须包含概念定义、核心区别、适用场景、工程取舍和常见误区。
* scenario_debug 题必须包含现象确认、可能原因、排查顺序、优化方案和验证指标。
* system_design 题必须包含核心模块、数据流、状态管理、异常处理、评估指标和安全边界中的多个方面。
* coding_or_pseudocode 题必须包含核心思路、伪代码、复杂度、边界情况和测试用例。
* 不允许只给一句空泛答案，例如“需要考虑性能、安全和稳定性”。

7. 难度标注必须合理

* difficulty 使用 1-5。
* 1：基础概念，知道定义即可。
* 2：能区分相似概念，理解基本流程。
* 3：能结合工程场景解释原因。
* 4：能排查问题、设计方案、分析取舍。
* 5：能做系统设计、复杂故障归因、指标评估或安全边界设计。
* 不允许所有题都标成同一难度。

8. knowledge_points 要可复用

* 每题建议 2-4 个 knowledge_points。
* knowledge_points 不能过大，例如只写“Agent”。
* knowledge_points 不能写成完整句子。
* 推荐写成可复用标签，例如“Agent 状态管理”“工具结果写回”“RAG 评估”“结构化输出校验”。

9. interview_followups 必须有面试价值

* 每题至少 2 个 interview_followups。
* 追问必须具体、可继续追问、贴近真实工程场景。
* 不允许使用“你怎么看？”“请展开说说？”这类空泛追问。
* 追问应能考察候选人的工程理解、项目经验、排障能力或系统设计能力。

10. 内容不能过于浅显

* 不要大量生成纯定义题。
* 每个模块都必须包含一定比例的工程场景题、排障题和系统设计题。
* 题目要服务于真实面试表达，而不是只服务于死记硬背。

11. 不得自相矛盾

* 题干、选项、answer、option_analysis、reference_answer、scoring_points 之间不能互相冲突。
* 如果题目引用某个工程原则，解析必须与该原则一致。
* 不允许出现题干问 A、答案解释 B 的情况。

* context compaction
* working set
* 长期记忆召回
* TTL
* 冲突检测
* token 成本

scoring_points:

* 能从上下文长度、历史注入和记忆召回入手定位问题。
* 能识别过期信息、重复摘要和工具长输出对回答质量的影响。
* 能提出 working set、摘要压缩、长期记忆检索和 TTL 策略。
* 能给出延迟、成本、冲突率等治理指标。

diagnosis_steps:

* 统计线上请求的平均 token 数、历史轮数、工具输出长度和长期记忆注入数量。
* 抽样检查上下文中是否存在重复、过期、冲突或低价值内容。
* 通过 trace 关联错误回答与被注入的历史或记忆条目。
* 区分问题来自模型推理、记忆召回、摘要压缩还是工具结果污染。
* 对高成本会话进行 replay，比较压缩前后的质量、延迟和成本变化。

optimization_points:

* 保留最近 n 轮作为 working set，旧历史进入摘要层。
* 长期记忆按相关性、时间衰减和可信度排序召回。
* 对过期偏好和事实设置 TTL、版本号和冲突覆盖策略。
* 工具长输出只保存结构化摘要、关键字段和引用指针。

def run_agent_loop(model, tool_registry, messages, max_steps=8, require_approval=None):
require_approval = require_approval or set()
seen_calls = set()

```
  for step in range(max_steps):
      action = model.generate(messages)

      if action.type == "final":
          return {
              "status": "ok",
              "answer": action.content,
              "steps": step + 1
          }

      if action.type != "tool_call":
          return {
              "status": "error",
              "reason": "unknown_action",
              "steps": step + 1
          }

      tool_name = action.tool_name
      args = normalize_args(action.args)

      if tool_name not in tool_registry:
          messages.append({
              "role": "tool",
              "name": tool_name,
              "content": "工具不存在，请重新规划。"
          })
          continue

      dedup_key = (tool_name, stable_json_dumps(args))
      if dedup_key in seen_calls:
          messages.append({
              "role": "tool",
              "name": tool_name,
              "content": "重复调用已拦截，请基于已有结果总结或重新规划。"
          })
          continue

      seen_calls.add(dedup_key)

      if tool_name in require_approval:
          approval_id = create_approval_request(tool_name, args)
          return {
              "status": "paused_for_approval",
              "approval_id": approval_id,
              "tool": tool_name,
              "args": args
          }

      try:
          result = tool_registry[tool_name](args)
      except Exception as e:
          result = {
              "ok": False,
              "error": str(e)
          }

      messages.append({
          "role": "tool",
          "name": tool_name,
          "content": serialize_result(result)
      })

  return {
      "status": "max_steps_exceeded",
      "steps": max_steps
  }
```

space: O(T + H)，其中 T 为最多循环步数，H 为 messages 历史和 seen_calls 去重集合占用。

stem: |
请写一个 Python 风格的记忆检索与压缩函数：保留最近 n_recent 条消息；从长期记忆中按“相似度 + 时间衰减”选出 top-k；当上下文超过预算时自动压缩早期消息。

reference_answer: |
核心思路是把上下文构造分为三层：最近消息、长期记忆召回和预算控制。

最近 n_recent 条消息作为 working memory，保证当前任务连续性。长期记忆通过相似度和时间衰减综合打分，避免只召回语义相似但已经过期的信息。使用小顶堆维护 top-k，避免对全部记忆排序带来的额外成本。

构造完成后，需要用 token_budget 控制上下文大小。如果超出预算，就优先压缩较早、价值较低的消息，保留摘要而不是简单丢弃全部信息。

answer_keywords:

* recent window
* long-term memory
* cosine similarity
* recency decay
* heap top-k
* token budget
* summarize

scoring_points:

* 能保留最近 n_recent 条消息作为 working memory。
* 能对长期记忆按相似度和时间衰减综合打分。
* 能用 top-k 选择最相关记忆。
* 能在超过 token_budget 时执行压缩。
* 能说明时间复杂度和空间复杂度。

interview_followups:

* 时间衰减权重过高会带来什么问题？

pseudocode: |
import heapq
import math

def score_memory(query_vec, memory):
similarity = cosine_similarity(query_vec, memory["vector"])
recency = math.exp(-0.05 * memory["age_days"])
confidence = memory.get("confidence", 1.0)
return 0.75 * similarity + 0.20 * recency + 0.05 * confidence

def build_context(recent_msgs, long_term_memories, query_vec, token_budget, n_recent=8, k=5):
working_set = recent_msgs[-n_recent:]

```
  heap = []
  for memory in long_term_memories:
      score = score_memory(query_vec, memory)
      item = (score, memory["id"], memory)

      if len(heap) < k:
          heapq.heappush(heap, item)
      elif score > heap[0][0]:
          heapq.heapreplace(heap, item)

  retrieved = [
      memory
      for score, memory_id, memory in sorted(heap, key=lambda x: x[0], reverse=True)
  ]

  context = []
  for msg in working_set:
      context.append(msg)

  for memory in retrieved:
      context.append({
          "role": "memory",
          "content": memory["text"],
          "memory_id": memory["id"]
      })

  while estimate_tokens(context) > token_budget and len(context) > 2:
      oldest = context.pop(0)
      summary = summarize_item(oldest)
      context.insert(0, {
          "role": "summary",
          "content": summary,
          "source": "context_compaction"
      })

      if estimate_tokens(context) > token_budget and context[0]["role"] == "summary":
          context[0]["content"] = shorten_summary(context[0]["content"])

  return context
```

complexity:
time: O(m log k + c)，其中 m 为长期记忆数量，k 为召回数量，c 为压缩过程中估算 token 和摘要处理的成本。
space: O(k + n_recent + c)，其中 k 为长期记忆召回数量，n_recent 为最近消息数量，c 为压缩后上下文占用。
